## 一、项目概述

### 1.1 项目定位

Feed Engine 是全球首个**去中心化人工预言机网络**，通过人工验证机制为链上协议提供可信的结算级价格数据。

**定位**：公共基础设施，不仅服务 NST 协议，也对外开放给其他第三方协议调用。

### 1.2 核心理念

喂价**不是**简单的"报告价格"，而是一个**结算责任层**。

系统解决三个根本问题：
1. **谁确认使用哪个价格？** → 喂价员（Feeder）
2. **谁为该价格负责？** → 签名者承担责任
3. **谁保证该价格的链上结算有效性？** → 质押担保机制

**核心原则**：API 市场数据 ≠ 有效结算价格。只有经过喂价员签名提交的价格，才能成为不可否认的、结算级别的链上价值。

### 1.3 解决的五大核心问题

| 问题 | 解决方案 |
|------|---------|
| 价格来源 | 喂价员使用自有市场工具验证，人工防止报价错误、暂停、缺口、异常 |
| 价格责任 | 签名 = 责任，准确则奖励，错误则惩罚 |
| 公平性 | 拍卖式抢单 + 多喂价员共识（2/3、3/5等）防止操纵 |
| 去中心化 | 平台不提供价格，不承担结算责任，所有责任由喂价员承担 |
| 激励机制 | 代币奖励 + 质押惩罚，保持喂价员诚实和长期参与 |

### 3.3 Gas 优化策略

**客户重点关注**：如何节省 Gas 费

| 优化方向 | 具体措施 |
|---------|---------|
| 批量操作 | 多个喂价打包成一笔交易提交 |
| 链下计算 | 共识计算在链下完成，只将结果上链 |
| 存储优化 | 使用 IPFS 存储订单详情，链上只存哈希 |
| 数据压缩 | 价格使用定点数存储，减少存储空间 |
| 事件替代存储 | 非关键数据通过事件记录而非状态存储 |
| 代理模式 | 使用 EIP-1167 最小代理降低部署成本 |
| 签名聚合 | 多签名聚合减少验证成本 |

```solidity
// Gas 优化示例：批量提交价格
contract FeedEngineBatch {
    // 批量提交价格哈希，节省多次调用的 Gas
    function batchSubmitPriceHash(
        bytes32[] calldata orderIds,
        bytes32[] calldata priceHashes
    ) external {
        require(orderIds.length == priceHashes.length, "Length mismatch");
        for (uint i = 0; i < orderIds.length; i++) {
            _submitPriceHash(orderIds[i], priceHashes[i]);
        }
    }

    // 使用 calldata 而非 memory 节省 Gas
    function submitPriceHash(
        bytes32 orderId,
        bytes32 priceHash
    ) external {
        _submitPriceHash(orderId, priceHash);
    }
}
```

## 四、喂价员体系设计

### 4.1 入场要求

喂价员必须满足以下条件：

1. **专业能力**：了解所选市场，选择熟悉的**国家**和**标的资产类型**
2. **数据来源**：能够获取官方市场数据
3. **质押担保**：提供以下**任一**形式的质押（三选一）：
   - ✅ FEED 代币
   - ✅ USDT 稳定币
   - ✅ 喂价员执照 NFT（可转让席位模式）

### 4.2 喂价员注册信息

## 五、智能订单分配系统

### 5.1 核心匹配逻辑

**客户需求**：根据喂价员选择的国家、交易所，智能分配待喂价订单。

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        智能订单分配流程                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  1. 订单进入系统                                                         │
│     └─→ 解析订单属性：国家、交易所、标的类型、名义本金                    │
│                                                                          │
│  2. 匹配喂价员                                                           │
│     └─→ 筛选条件：                                                       │
│         ├─ 喂价员选择的国家包含该订单国家                                 │
│         ├─ 喂价员选择的交易所包含该订单交易所                              │
│         ├─ 喂价员选择的资产类型匹配                                       │
│         ├─ 喂价员等级满足订单要求                                         │
│         └─ 喂价员当日未超单日上限                                         │
│                                                                          │
│  3. 优先级排序                                                           │
│     └─→ 排序因子：                                                       │
│         ├─ 历史准确率（权重 40%）                                         │
│         ├─ 响应速度（权重 30%）                                           │

│                                                                          │
│  4. 推送订单                                                             │
│     └─→ 向匹配的喂价员推送订单卡片                                        │

│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 5.2 订单可见性规则

**客户确认**：没有抢中订单的喂价员也能看见订单信息。

| 状态 | 抢中的喂价员 | 未抢中的喂价员 |
|-----|------------|--------------|
| 订单详情 | ✅ 完整可见 | ✅ 可见（只读） |
| 提交喂价 | ✅ 可操作 | ❌ 不可操作 |
| 查看进度 | ✅ 实时更新 | ✅ 实时更新 |
| 查看结果 | ✅ 可见 | ✅ 可见 |

**设计意图**：
- 提高透明度，让所有相关喂价员了解市场动态
- 学习机会：新手可观察高手如何喂价
- 监督作用：公开透明减少作弊可能

### 6.2 订单类型定义

| 喂价类型 | 用途 | 责任级别 | 奖励倍数 |
|---------|------|---------|---------|
| 初始喂价 (Initial Feed) | 建立初始保证金参考价 | 中等 | 1.0x |
| 动态喂价 (Dynamic Feed) | 盘中追保/强平风险判断 | 高 | 1.5x |
| 结算喂价 (Settlement Feed) | 确定盈亏、行权价值、强平价 | 最高 | 2.0x |
| 仲裁喂价 (Arbitration Feed) | 争议案例处理（仅限A/S级喂价员） | 最高 | 3.0x |
